# Архитектура

*[Read in English](./architecture.md)*

## Обзор системы

WowFit работает как три контейнера, оркестрируемых через Docker Compose:

```
                        ┌───────────────────────────────┐
    Браузер  ─────────▶ │  frontend (nginx)                │
                        │  - отдаёт собранное React-приложение │
                        │  - проксирует /api, /docs, /redoc, │
                        │    /openapi.json на бэкенд          │
                        └───────────────────────────────┘
                                        │
                                        ▼
                        ┌───────────────────────────────┐
                        │  backend (FastAPI / Uvicorn)      │
                        │  роутер → сервис → модель данных    │
                        └───────────────────────────────┘
                                        │
                                        ▼
                        ┌───────────────────────────────┐
                        │  db (PostgreSQL 16)                │
                        └───────────────────────────────┘
```

Контейнер фронтенда собирается в два этапа: React-приложение компилируется с помощью Vite, а полученные статические файлы отдаются через nginx, который также проксирует API-трафик на контейнер бэкенда через внутреннюю сеть Docker — поэтому браузер всегда обращается только к одному источнику (origin), и на стороне клиента не требуется обработка CORS.

Деплой непрерывный: каждый пуш в ветку `main` запускает workflow GitHub Actions, который подключается к серверу деплоя по SSH и пересобирает стек Docker Compose, так что `main` — это всегда то, что работает в продакшене на [hi-baam.space](https://hi-baam.space/).

## Слои бэкенда

Бэкенд следует строгой многослойной структуре: `роутер → сервис → модель данных`. Роутеры остаются тонкими — они разбирают запрос, вызывают один метод сервиса и возвращают его результат. Вся бизнес-логика, валидация и оркестрация находятся в слое сервисов.

```
backend/app/
  core/
    database.py       Управление движком и сессиями SQLAlchemy
    mailer.py          Отправка почты по SMTP, с резервным вариантом логирования, когда SMTP не настроен
    scheduling.py       Вспомогательные функции для расчёта дат и стриков
    security/
      authHandler.py     Выпуск и проверка JWT (access, refresh и admin токены)
      googleVerifier.py   Проверка Google ID-токенов
      hashHelper.py        Хеширование паролей с помощью bcrypt
  db/
    models/            Модели SQLAlchemy ORM
    schema/             Pydantic-модели запросов и ответов
  routers/             По одному тонкому роутеру на предметную область
  service/             Бизнес-логика, по одному классу на предметную область
  util/
    init_db.py           Создание таблиц, миграции схемы и начальные данные при старте
    protectRoute.py        Зависимости FastAPI, обеспечивающие аутентификацию пользователей и администратора
```

## Аутентификация и авторизация

Пользователи могут аутентифицироваться по email и паролю или через вход с Google. Пароли хешируются с помощью bcrypt; аутентификация через Google проверяет подпись ID-токена, срок действия, издателя и получателя относительно настроенного client ID приложения в Google, и доверяет email только в том случае, если сам Google отметил его как подтверждённый.

Выпускается три отдельных типа JWT-токенов, каждый из которых содержит поле `type`, поэтому один тип не может быть подставлен вместо другого:

- **Access-токены** (30 минут) аутентифицируют все защищённые пользовательские маршруты.
- **Refresh-токены** (14 дней) принимаются только на маршруте обновления токена.
- **Admin-токены** (2 часа) аутентифицируют админ-панель, защищённую единым общим паролем, а не привязкой к аккаунту пользователя.

Забытый пароль восстанавливается с помощью 6-значного кода, отправляемого на почту, действительного 15 минут. Если исходящая почта не настроена, код вместо этого записывается в логи сервера, поэтому весь процесс остаётся полностью тестируемым без настройки SMTP.

## Основной игровой цикл: тренировки, стрики и таблицы лидеров

Отправка тренировки — это центральная операция записи во всей системе. При отправке тренировки:

1. Система проверяет, что участие пользователя и сам челлендж активны, что упражнение принадлежит этому челленджу, и что сегодня действительно запланированный день по расписанию этого челленджа.
2. В таблицу `sessions` добавляется неизменяемая запись — этот журнал никогда не редактируется и является итоговым источником истины для всего, что из него выводится.
3. Отправленные чистые повторения добавляются к прогрессу по этому упражнению за день. Как только суммарное число чистых повторений за день по упражнению достигает цели, это упражнение отмечается закрытым на этот день.
4. Как только все упражнения в челлендже закрыты на этот день, закрывается и сам день:
   - Счётчик выполненных дней участника увеличивается на единицу.
   - Его **стрик в рамках челленджа** продлевается на единицу, если предыдущий запланированный день (по собственному ежедневному или еженедельному расписанию этого челленджа) тоже был закрыт; в противном случае стрик сбрасывается до единицы.
   - **Общий стрик пользователя** — единый, основанный на календарных днях стрик, общий для всех челленджей пользователя — продвигается на единицу за этот календарный день, независимо от того, сколько челленджей или упражнений было выполнено.

Поскольку значения стриков обновляются только в момент фактического закрытия дня, пропущенный день не сбрасывает ничего в базе данных заранее — вместо этого актуальный, корректный стрик пересчитывается при каждом запросе профиля пользователя или недельного представления, поэтому пропущенный день всегда отображается точно, без необходимости в фоновой задаче.

Одновременные отправки для одного и того же участия защищены блокировкой на уровне строки, что гарантирует невозможность потери прогресса из-за race condition при одновременных отправках тренировок.

**Ранжирование таблицы лидеров** для конкретного челленджа упорядочивает участников по:
1. Стрику в рамках челленджа, по убыванию
2. Числу выполненных дней, по убыванию
3. Суммарному числу чистых повторений, по убыванию
4. Дате присоединения, по возрастанию (присоединившиеся раньше занимают более высокое место при полном равенстве)

Каждая строка таблицы лидеров также показывает общий стрик участника рядом с его стриком в рамках челленджа, поскольку они отражают разные вещи — ритм конкретного челленджа против общей регулярности пользователя во всём, в чём он участвует.

Когда отправка тренировки продвигает общий стрик, фронтенд ставит в очередь празднование: значок пламени стрика на главном экране проигрывает короткую анимацию огня при следующей отрисовке, а затем возвращается к статичному кадру покоя.

## Готовые челленджи (пресеты)

Чтобы приложение никогда не запускалось с пустым списком челленджей, десять готовых челленджей с разными упражнениями, целями и расписаниями создаются автоматически при каждом запуске бэкенда. Они принадлежат единому, автоматически создаваемому системному аккаунту и создаются сразу в публичном состоянии, минуя цепочку «приватный → публикация», через которую проходят обычные пользователи. При повторном запуске сидирования уже существующие пресеты обновляются на месте, а не дублируются, поэтому повторный деплой всегда безопасен.

## Жизненный цикл челленджа: архивирование, возобновление и истечение срока

Чтобы остаться в таблице лидеров, не обязательно держать челлендж активным — участник может лично заархивировать челлендж, что меняет только его собственное представление и никак не влияет на остальных участников или таблицу лидеров. Архивирование можно отменить в любой момент, возобновив участие.

Челлендж с истёкшим сроком действия (прошла его конечная дата) технически всё ещё можно возобновить на уровне API — бэкенд не применяет никакой проверки истечения срока при переводе участия обратно в активное состояние. Фронтенд обеспечивает это ограничение самостоятельно, скрывая или отключая действие «возобновить» после того, как конечная дата челленджа прошла, — то есть практическое ограничение на данный момент реализовано на стороне клиента, а не в самом API.

## Конвейер компьютерного зрения на стороне клиента

Проверка выполнения упражнений происходит полностью на клиенте. Фронтенд загружает модель MediaPipe Pose Landmarker во время работы и обрабатывает поток с веб-камеры локально в браузере — видео никогда не загружается на бэкенд. Углы суставов тела вычисляются по обнаруженным точкам позы на каждом кадре и передаются в конечный автомат для каждого упражнения, который считает приседания, отжимания и удержание планки, обеспечивая живую визуальную обратную связь и звуковые сигналы при подтверждении повторений. На мобильных устройствах вид камеры фиксируется в соотношении сторон 3:4 и учитывает ориентацию устройства, чтобы модель позы всегда получала стабильно кадрированный поток.

Пороговые значения для подсчёта повторений были откалиброваны в отдельном прототипе на наборе размеченных видео с отжиманиями, прежде чем попасть в продакшн-приложение. По завершении тренировки на бэкенд отправляется только её итоговый результат — общее число повторений, число чистых повторений и длительность, — и бэкенд считает этот переданный результат достоверным; у бэкенда нет доступа к самому видео или покадровым данным позы.

## Учёт шагов

Данные о шагах за день хранятся в единой таблице, заполняемой из двух независимых источников:

- **Withings**, через стандартный поток подключения OAuth2. После подключения бэкенд периодически обновляет токен доступа по мере необходимости и забирает недавние суммарные данные о шагах из API активности Withings.
- **Компаньон-приложение для смартфона**, которое читает данные о шагах из Android Health Connect или Apple HealthKit и передаёт их напрямую на бэкенд. Это продолжает работать для пользователей без устройства Withings.

Оба источника записывают данные в одну и ту же таблицу шагов, различаясь по происхождению, поэтому остальной части приложения — в частности, виджету шагов в профиле — не нужно знать или заботиться о том, из какого источника поступили данные за конкретный день.

## Мобильный интерфейс

Поверх общего бэкенда и API фронтенд включает отдельный мобильный интерфейс: собственный поток онбординга, мобильные макеты для главного экрана, ленты контента и просмотра челленджей (навигация каруселью, вкладки и плавающая кнопка действия), а также мобильный экран тренировки с фиксированным соотношением сторон камеры и обработкой ориентации устройства — всё это адаптировано под условия тренировки со смартфоном в руке. Анимация празднования стрика и обновлённое оформление компонентов (значки, виджет стрика) применяются как на десктопе, так и на мобильных устройствах.
